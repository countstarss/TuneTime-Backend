# 订单履约补救一期后端方案

本文是 TuneTime 后端针对“订单履约异常补救一期”的实施文档。

目标只有一个：

- 先把订单从接单、支付、到达、上课、完课、评价、争议这一整条流程跑顺。
- 暂时不做真实资金结算、钱包冲正、微信零钱提现。
- 但要先把“结算资格”和“异常阻断”这层语义补齐，避免后面接资金时重做状态机。

相关来源文档：

- [TuneTime-WX/docs/order-lifecycle-scenario-matrix.md](/Users/luke/TuneTime/TuneTime-WX/docs/order-lifecycle-scenario-matrix.md)

## 1. 一期范围

本期后端新增和收口的内容如下：

1. 给 `lessons` 和 `teacher-reviews` 补充鉴权和角色控制。
2. 给订单补充完课确认状态、异常状态、结算资格状态。
3. 新增预约异常工单 `BookingExceptionCase`。
4. 新增课程证据 `LessonEvidence`。
5. 新增老师到达、家长确认完课、家长发起争议、后台处理争议、后台人工修复接口。
6. 新增自动巡检任务，处理超时关闭、逾期未签到、逾期未签退、自动确认完课。

本期明确不做：

- 真实分账
- 钱包入账/冲正
- 微信打款
- 部分退款资金流

## 2. 新增状态语义

### 2.1 Booking 新增字段

- `completionStatus`
  - `PENDING_TEACHER_RECORD`
  - `PENDING_GUARDIAN_CONFIRM`
  - `GUARDIAN_CONFIRMED`
  - `AUTO_CONFIRMED`
  - `DISPUTED`
- `completionConfirmedAt`
- `exceptionStatus`
  - `NONE`
  - `OPEN`
  - `BLOCKING`
- `settlementReadiness`
  - `NOT_READY`
  - `READY`
  - `BLOCKED`

说明：

- `completionStatus` 解决“老师签退了，但家长还没确认”这一段流程缺失的问题。
- `exceptionStatus` 解决“订单是否处于异常处理中”的问题。
- `settlementReadiness` 只是资格标记，本期不直接驱动真钱流转。

### 2.2 BookingExceptionCase

用于承接异常单，不再让异常状态只存在前端判断里。

核心字段：

- `bookingId`
- `exceptionType`
- `status`
- `responsibilityType`
- `summary`
- `resolution`
- `metadata`
- `createdByUserId`
- `resolvedByUserId`

当前支持的异常类型：

- `PAYMENT_MISMATCH`
- `OVERDUE_NOT_STARTED`
- `OVERDUE_NOT_FINISHED`
- `ARRIVAL_DISPUTE`
- `NO_FEEDBACK`
- `LESSON_QUALITY_DISPUTE`
- `DURATION_DISPUTE`
- `MANUAL_REPAIR_REQUIRED`

### 2.3 LessonEvidence

课程证据单独建模，避免把所有证据都塞进单个视频字段。

当前支持：

- `ARRIVAL_PHOTO`
- `ARRIVAL_VIDEO`
- `RESULT_PHOTO`
- `RESULT_VIDEO`
- `OTHER`

## 3. 新增接口

### 3.1 订单接口

- `POST /bookings/:id/arrival`
  - 老师确认到达
- `POST /bookings/:id/complete-confirm`
  - 家长确认完课
- `POST /bookings/:id/disputes`
  - 家长发起争议
- `POST /bookings/:id/disputes/:caseId/resolve`
  - 后台处理争议
- `POST /bookings/:id/ops/manual-repair`
  - 后台人工修复

### 3.2 课程接口

- `POST /lessons/:id/evidences`
  - 上传课程证据链接

### 3.3 权限收口

本期把以下接口从“无保护”改成“必须鉴权”：

- `lessons`
- `teacher-reviews`

策略：

- 老师只能操作自己名下订单对应的课程
- 家长只能访问和评价自己名下订单
- 管理员可以执行人工修复和争议处理

## 4. 状态推进规则

### 4.1 到达

- 老师调用 `arrival` 后，只记录到达事实，不直接推进到上课中。
- 真正开课仍由 `check-in` 推进到 `IN_PROGRESS`。

### 4.2 签退

- 老师签退后：
  - `booking.status = COMPLETED`
  - `completionStatus = PENDING_TEACHER_RECORD`
  - `settlementReadiness = NOT_READY`

### 4.3 提交课后反馈

- 老师提交课后反馈后：
  - `completionStatus = PENDING_GUARDIAN_CONFIRM`
  - `settlementReadiness = NOT_READY`

### 4.4 家长确认完课

- 家长确认后：
  - `completionStatus = GUARDIAN_CONFIRMED`
  - `completionConfirmedAt` 写入
  - 若无阻断异常，`settlementReadiness = READY`

### 4.5 发起争议

- 家长发起争议后：
  - 新增或更新 `BookingExceptionCase`
  - `completionStatus = DISPUTED`
  - `exceptionStatus = BLOCKING`
  - `settlementReadiness = BLOCKED`

### 4.6 后台处理争议

- 管理员处理争议后：
  - 写入责任方和处理结论
  - 若没有其他未关闭异常，可恢复 `exceptionStatus`
  - 是否恢复 `settlementReadiness` 由处理结果决定

## 5. 后台人工修复动作

`POST /bookings/:id/ops/manual-repair` 当前支持：

- `PAYMENT_STATUS`
  - 手工补支付状态
- `CHECK_IN`
  - 手工补签到
- `CHECK_OUT`
  - 手工补签退
- `RESPONSIBILITY`
  - 手工改责任方
- `CLOSE_EXCEPTION`
  - 手工关闭异常工单
- `COMPLETION_STATUS`
  - 手工修正完课确认状态

所有人工修复都会写入 `AdminAuditLog`。

## 6. 自动巡检任务

当前后端在模块启动后会启动轻量巡检：

1. `PENDING_ACCEPTANCE` 超时自动关闭
2. `PENDING_PAYMENT` 超时自动关闭
3. 已过上课时间仍未签到时，自动创建 `OVERDUE_NOT_STARTED`
4. 已过下课时间仍未签退时，自动创建 `OVERDUE_NOT_FINISHED`
5. 已完课且老师已提交反馈，家长超时未处理时，自动改成 `AUTO_CONFIRMED`

默认配置：

- 待接单超时时间：`BOOKING_PENDING_ACCEPTANCE_TTL_MINUTES`
  - 默认 `720`
- 自动确认完课小时数：`BOOKING_AUTO_CONFIRM_HOURS`
  - 默认 `48`
- 是否启用巡检：`BOOKING_LIFECYCLE_AUTOMATION_ENABLED`
  - 默认启用

## 7. 本期仍然保留的后续项

虽然本期不做真钱流转，但下面这些能力已经为二期预留了接口语义：

- `settlementReadiness`
- 异常阻断
- 责任方判定
- 证据留存
- 后台审计日志

后续二期可以在不推翻当前状态机的前提下继续补：

- 结算单
- 钱包流水
- 部分退款
- 提现限制
- 已提现后的追缴或后续单抵扣

## 8. 落地建议

部署顺序建议：

1. 先执行 Prisma schema 同步和客户端生成。
2. 再部署后端服务。
3. 最后再让小程序和管理后台接新接口。

如果管理后台还没接完：

- 先开放老师端到达、签到、签退、提交反馈、上传证据
- 争议处理和人工修复先通过 Swagger 或内部后台调用

这样可以先把流程跑顺，再逐步补运营侧界面。
